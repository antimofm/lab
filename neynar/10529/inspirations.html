Twenty in-app browser / mini-app chrome references surveyed for the Neynar redesign. The eight archetypes on the main page are distilled from these. Clicking each chip opens the source.

Source: NEYN-10529 · execution: CLIENTS-38

Crypto wallets mini-app + dapp browsers

  • Rainbow ↗

    Tall multi-row chrome: URL/search field with tab-switcher, plus a horizontal trending-dapps strip above content. Browser as destination, not transit.

    Anti-reference for Neynar — leans into chunky because the browser is the value. Neynar's brief is the opposite.

  • MetaMask ↗

    Two-row chrome: account/network selector pill on top, URL/search bar with magnifying glass + tabs + overflow below. Connection state and chain surfaced inline rather than hidden.

    Density-rich; example of how much state a wallet thinks it has to surface.

  • Coinbase Wallet ↗

    Single-row top bar: URL/title + secure padlock + tabs + overflow. Tabs treated like Safari (separate carousel). Chain switching happens silently in background, freeing chrome real estate.

    Closest ancestor to Base App's chrome.

  • Trust Wallet ↗

    Standard browser-like: back arrow, URL/title, share, overflow. Multichain selector behind the menu. Generic mobile browser dropped into the app shell.

    Neutral baseline.

  • Phantom ↗

    Compact single-row: URL/title centered, close (X) on left, share/overflow on right. Network indicator (Solana/EVM/Bitcoin) implicit — no chain pill in chrome. Among the lightest wallet chromes in this set.

    Strong reference for "minimal wallet chrome" — closest aesthetic to where Neynar likely wants to land.

  • Zerion ↗

    URL/search bar at top with avatar of active wallet (identicon) on left for fast multi-account switching, bookmark/star, tabs and overflow on right. Identity is the leftmost affordance, not back.

    Putting identity (not navigation) first is a Farcaster-shaped idea worth borrowing.

  • Uniswap Wallet ↗

    Lean single-row bar: URL or dapp title, network pill, close. Dapp browser feels like a sub-mode rather than peer surface — chrome stays out of the way.

    Good "browser as guest mode" reference.

  • Base App ↗

    Mini-app surface: top header with app name + author handle + close (X), bottom-anchored action affordances. Mini-app chrome is minimal but the embedding feed surface adds a thick parent header above it — net visual weight is high.

    Direct competitor surface — exactly the chrome Neynar is being compared against.

Social apps in-app browsers

  • X (Twitter) ↗

    Top bar: done/close + site title + URL stacked + share + overflow. Bottom bar: back/forward/share/Safari-handoff. Two-bar layout — chunky for "just open the link" use case.

    Anti-pattern: two bars where one would do.

  • Instagram ↗

    Tall double-bar: × + stacked title/URL + open-in-Safari + overflow on top; back/forward/share on bottom. Brand-grey with thick separators. Famously chunky — eats ~120pt before any content shows.

    The canonical anti-pattern. Pin to the moodboard as "do not do this."

  • TikTok ↗

    Single top bar: × + truncated title (max 10–20 chars) + overflow with "open in browser" / share. Compact but truncation is aggressive — title often unreadable.

    Solves chunkiness by truncating hard. Tradeoff: title legibility.

  • Snapchat ↗

    Branded yellow accents: × + title + share + overflow on top; back/forward + "open in Safari" on bottom. Less heavy than Instagram but still two-bar.

  • Reddit ↗

    Single top bar: close + title/URL + overflow; sometimes adds a "comments" affordance to jump back to the post. Sticky "view comments" bar returns to thread context — distinctive contextual return path.

    "Return to context" affordance — mini-apps could borrow this for "return to cast."

  • Discord ↗

    Stock WKWebView wrapper on iOS: close + URL + share + open-in-Safari; minimal Discord branding. Generic and forgettable.

    Reference for "lazy webview wrapper" — what NOT to ship if chrome is meant to feel native.

  • Warpcast ↗

    Per spec: header above the mini-app showing app name + author, plus a close (X). In practice the rendered header stacks below Warpcast's own nav, making total vertical chrome ~2 bars. The "too chunky" surface the redesign is targeting.

    The thing being beaten. Dual-header (host nav + mini-app header) is the core problem.

Mini-app paradigms canonical references

  • Telegram Mini Apps ↗

    Default: thin Telegram header with app title, back/close, overflow menu. Mini Apps 2.0 introduced full-screen mode that hides BOTH Telegram top and bottom bars — most aggressive minimization in this set.

    Strongest reference. Full-screen escape hatch is exactly the lever Neynar should consider.

  • WeChat Mini Programs ↗

    Iconic "capsule" menu: a single floating pill in the top-right with two buttons — overflow (•••) and close (○). No URL bar at all. Title is the page title set by the mini-program.

    The OG. The capsule pattern is probably the single highest-leverage reference for Neynar — minimum chrome, maximum app surface.

  • LINE Mini Apps ↗

    Platform-generated header with mandatory action button (cannot be hidden by developer); shows domain (unverified) or App Name + verified badge (verified). Redesigned late 2024 — colors no longer customizable, enforcing visual consistency.

    Useful precedent for "platform owns the chrome, app cannot override" — a stance Neynar/Farcaster could take.

System primitives baselines

  • SFSafariViewController (iOS) ↗

    Top: Done + site title above small URL with padlock + reload/share. Bottom: back/forward/share/tabs/Safari-handoff. App can tint the bar but not restructure. iOS 14+ adds collapse-on-scroll. Reader mode and AutoFill come free.

    Baseline truth. Anything heavier in a host app is a deliberate (often costly) choice.

  • Chrome Custom Tabs ↗

    Single top bar: × + site title (or URL) + optional brand action button + overflow. Toolbar color themable per scheme. setUrlBarHidingEnabled() collapses bar on scroll. Picture-in-picture added 2024.

    Cleaner than iOS equivalent — single bar, hides on scroll. Strong baseline for what "good default" looks like.
