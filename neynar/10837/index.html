Network Agent UX

Three UX directions for an agent that helps users grow and navigate their Farcaster network — from ambient inline nudges to a dedicated copilot surface.

Farcaster client (mobile + web). Casts, threads, feed, channels.

Global agent (shared), not personal-per-user agents.

Protocol-level agent comms. Warpcast-specific chrome. Backend architecture.

Native, not bolted on. Contextual at moment of action.

Overview

Three directions

Each direction takes a different position on where the agent lives — from invisible (ambient) to explicit (tab) to conversational (drawer). The right answer may combine elements from each.

Direction A

Ambient Intelligence

The agent has no home — it shows up inline within the existing feed and thread experience. Suggestions appear at the moment of relevance, then disappear. No new surface.

Most native Zero friction Harder to discover Low agency feel
Direction B Recommended primary

Network Copilot Tab

A dedicated fifth tab acts as the agent's home — a network dashboard with daily picks, intros, events, and group chats. Ambient nudges still appear inline. Best of both.

Discoverable High agency Permission-friendly Adds chrome
Direction C

Conversational Drawer

A persistent floating button opens a slide-up chat drawer. Context-aware: the agent knows what screen you're on. Flexible but risks feeling like a support widget.

Flexible Conversational Intrusive Less native feel
Direction A Ambient

Ambient Intelligence

The agent lives entirely within existing surfaces. It surfaces a suggestion when the context is right and disappears when it isn't. Onboarding happens as a cast from the agent's FID. Thread-level nudges appear as an inline card between replies. Discovery cards appear in the feed like promoted content, but earned.

Architecture decision

Global agent vs Personal agents

The brief leans global. Below is the full tradeoff matrix — then the recommendation.

Dimension Global agent Personal agents (P2P)
Implementation complexity Low — one model, one surface High — agent-to-agent protocol, trust, handoffs
Personalisation feel Medium — shared model, per-user context High — owned by each user
Trust / permission model Simple — one entity, clear permissions surface Complex — who does my agent trust? What can theirs read?
Intro brokering Direct — single agent sees both users Indirect — requires agent negotiation, latency
Permission justification Easy — feels like granting access to "your assistant" Confusing — why does a third party need my calendar?
Failure modes Single point of failure, but recoverable Cascade failures; message loss across agent boundary
Creepiness ceiling Moderate — "who told it that?" High — "your agent told my agent what?"
Alignment with "Meta glasses" vision Strong — one ambient intelligence Weak — agents talking to each other feels mechanical
Scale to millions of users Standard API scaling Exponential agent-to-agent communication surface

✓ Global agent — strengths

  • One model with full network visibility brokers intros faster and more accurately
  • Permission surfaces are legible: you're granting access to "your guide", not a third-party agent
  • Feels personal because it's contextualised to you, even though it's shared
  • Matches existing user mental model (like having a well-connected friend)
  • Simpler to build, test, and iterate

Personal agents — when it might matter

  • If users want strong data ownership (agent only knows what I told it)
  • If the protocol moves toward verifiable agent credentials
  • For power users who want to configure their own agent behavior
  • Long-term: personal agents could be wallets that negotiate access
  • Could be a v3 feature, not a v1 architecture constraint
Recommendation

Primary direction

Direction B (Copilot Tab) with Direction A inline ambient layer — backed by a Global agent

The copilot tab gives the agent a legible home — somewhere users can go intentionally, review suggestions, and understand what the agent has access to. This is critical for trust and for justifying deeper permissions like calendar and wallet. The tab answers "where does this live?" clearly: it has its own space in the nav, it's always there when you need it.

The ambient layer (Direction A) runs on top of this — thread-level reply suggestions and feed-level discovery cards appear at the moment of relevance. They don't require the user to go looking. These feed back into the copilot tab when dismissed, so nothing is lost.

Direction C (drawer) works best as a secondary power-user entry point, not the primary surface. It can live inside the copilot tab as a chat mode for when someone wants to ask the agent something directly.

The global agent model is the clear choice at this stage. It's lower complexity, more legible for permissions, and matches the "personal assistant with a giant network" metaphor better than agents negotiating with each other. Personal agents can be explored as a v3 trajectory if user ownership becomes a product differentiator.

Evolution

From onboarding help to ambient intelligence

The brief asks how this evolves over time. Three phases — each unlockable, each additive.

Phase 1 · Now

Network starter

  • Onboarding interest capture
  • Who to follow recommendations
  • Thread reply suggestions
  • Basic event discovery
  • No permissions required
Phase 2 · +Permissions

Active copilot

  • Intro facilitation (opt-in)
  • Calendar integration
  • Event organisation
  • Group chat matching
  • Wallet-gated group access
Phase 3 · Ambient

Real-time guidance

  • Proactive contextual nudges
  • In-thread live suggestions
  • Event organisation from the product
  • Network health score
  • "Meta glasses" ambient layer
